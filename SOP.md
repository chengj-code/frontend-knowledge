# 前端知识文档生成标准流程（SOP）

> 本文档定义了为每个前端模块生成基础知识指南和面试题库的完整工作流。
> 目标：**保证每个模块的文档质量稳定在 9.0+ 分**。

---

## 二、Phase 1：文档生成规范

### 2.1 基础知识指南模板

文件路径：`{模块名}/basics/{模块名}-Basics-Guide.md`

#### 必含章节（按模块特点裁剪，通常 12-18 章）

| 序号 | 章节类别 | 说明 |
|------|----------|------|
| 第1章 | **概述** | 是什么、核心特性、版本演进、安装方式 |
| 第2章 | **基础语法/类型** | 最基础的概念入口 |
| 第3-6章 | **核心机制** | 该领域最重要的 3-4 个核心概念（如 JS 的原型链/闭包/异步；Vue 的响应式/组件/路由） |
| 第7-10章 | **进阶特性** | 中高级功能 |
| 第11-13章 | **工程化/生态** | 构建工具、周边生态集成 |
| 第14章+ | **最佳实践** | 编码规范、常见陷阱、性能优化 |
| 附录A | **补充内容** | 版本对比、旧版兼容等 |
| 附录B | **实战案例** | 贯穿全文档的项目代码 |
| 附录C | **新特性速查** | 最新版本的 feature 汇总 |

#### 代码规范要求

```markdown
// ✅ 每个关键行都要有中文注释
const count = ref(0)        // 响应式数字，初始值为 0
const double = computed(() => count.value * 2)  // 计算属性：count 的双倍值

// ❌ 不要出现无注释的关键代码
const x = arr.map(i => i * 2)

// ✅ 复杂逻辑要有分步骤注释
// Step 1: 过滤掉空值
const filtered = items.filter(item => item != null)
// Step 2: 对剩余元素做转换
const result = filtered.map(item => transform(item))
```

#### 图示要求

- **必须包含至少 3 个 ASCII/Mermaid 图**：
  1. 核心机制的**工作流程图**
  2. 关键概念的**关系图/对比表**
  3. 数据结构的**内存模型图**

### 2.2 面试题库模板

文件路径：`{模块名}/interviews/{模块名}-Interview-Questions.md`

#### 题目数量与分布

| 层级 | 数量占比 | 难度 | 特征 |
|------|----------|------|------|
| 基础层 ★☆☆ | ~30% | 概念记忆、基础语法 | 答案直接明确 |
| 进阶层 ★★☆ | ~40% | 多点综合、需理解原理 | 需要分析推理 |
| 专家层 ★★★ | ~30% | 源码级深度、架构设计 | 手写实现、方案设计 |

**总题数：45-55 道**（根据模块复杂度调整）

#### 每道题格式

```markdown
## QXX: [题目内容]
- **难度**：★☆☆ / ★★☆ / ★★★
- **知识点**：[标签1] / [标签2]
- **题型**：选择题 / 简答题 / 代码分析题 / 编程实践题

### 参考答案要点：

1. **要点一**：[具体说明]
   ```代码示例（如有）```
2. **要点二**：[具体说明]
   ...
```

#### 必须覆盖的考点类型

- 基础概念辨析（A vs B 区别）
- 工作原理（为什么/怎么做）
- 经典面试输出题（代码执行结果）
- 手写实现（核心 API 的简化版）
- 场景设计题（如何解决某类问题）
- 性能优化策略
- 工程化实践

## 三、Phase 2：质量评审打分卡

### 3.1 打分维度详解

| 维度 | 满分 | 扣分项 | 典型扣分值 |
|------|------|--------|-----------|
| **结构完整性** | 1.5 | 缺少重要章节（如无工程化/无最佳实践） | -0.3/章 |
| **代码质量** | 2.0 | 注释不足、代码不可运行、缺少边界处理 | -0.2~0.5 |
| **难度分层** | 1.5 | 梯度不合理、专家题偏简单 | -0.2~0.4 |
| **面试实用度** | 1.5 | 答案太泛、缺乏实际场景 | -0.2~0.3 |
| **深度** | 2.0 | 缺少手写实现、原理只讲表面 | -0.5~1.0 |
| **工程化** | 1.5 | 无实战案例、无配置示例 | -0.3~0.5 |

### 3.2 常见问题清单（自动检查项）

在打分时逐项检查：

**基础知识指南：**
- [ ] 目录是否完整且可跳转？
- [ ] 每章是否有"本章要点速查"总结表格？
- [ ] 代码示例是否都有行内注释？
- [ ] 是否有至少 3 张流程图/示意图？
- [ ] 是否有贯穿性的实战案例？
- [ ] 是否有新旧版本对比/迁移相关内容？

**面试题库：**
- [ ] 题目总数是否达到 45+ 道？
- [ ] 三层难度分布是否合理（3:4:3）？
- [ ] 核心原理题是否有手写实现代码？
- [ ] 是否有追问链设计？
- [ ] 是否有知识体系速查表附录？
- [ ] 选择题是否有解析？编程题是否有参考答案？

### 3.3 评分等级参考

| 评分 | 等级 | 含义 | 后续动作 |
|------|------|------|----------|
| 9.0-10 | **优秀** | 可直接发布 | 仅做微调（可选） |
| 8.0-8.9 | **良好** | 需要优化 | 执行 P0-P2 优化 |
| 7.0-7.9 | **合格** | 需要较大改进 | 执行 P0-P4 全部优化 |
| <7.0 | **不合格** | 需要重构 | 重新生成或大幅补写 |

## 四、Phase 3：优化操作清单

### 4.1 P0：核心原理题手写实现（必做）

**目标题目**：涉及"底层原理"的进阶/专家题

**操作方式**：在答案末尾追加 `### 深度拓展：手写实现` 小节

**必须手写的典型主题**：
- 核心 API 的内部实现（如 Promise / Event Loop / 响应式系统 / diff 算法）
- 关键函数的手写版（call / apply / bind / new / instanceof / debounce / throttle / deepClone）
- 数据结构的手写版（LRU 缓存 / 发布订阅 / 并发调度器）

**代码要求**：
- 每行关键代码有注释
- 包含使用示例
- 用伪代码/ASCII 解释算法思路

### 4.2 P1：流程图/示意图（必做）

**插入位置**：对应章节的核心概念处

**图表类型分配**：
| 图表类型 | 适用场景 | 格式 |
|----------|----------|------|
| 流程图 | 工作机制、执行流程 | Mermaid flowchart |
| 时序图 | 生命周期、调用顺序 | Mermaid sequenceDiagram |
| 决策树 | 方案选择 | Mermaid flowchart |
| 集合图 | 类型关系（联合/交叉） | ASCII Venn 图 |
| 结构图 | 数据模型、内存布局 | ASCII 树形图 |
| 转换图 | 类型变换过程 | ASCII 映射图 |
| 步骤演示 | 算法执行过程 | ASCII 逐步展示 |

**最低要求**：基础知识指南 ≥ 4 张图，面试题库 ≥ 2 张图

### 4.3 P2：追问链设计（强烈推荐）

**目标题目**：核心面试题（约 15-20 道，覆盖各难度层级）

**格式**：
```markdown
### 🔍 追问链
1. **追问标题**
   → 方向：答题要点提示
```

**追问设计原则**：
- 从基础概念 → 原理深度 → 工程实践递进
- 每题 2-4 个追问
- 追问方向要精炼有用

### 4.4 P3：实战案例（推荐）

**要求**：
- 一个完整的、可运行的项目
- 串联该模块 60% 以上的核心知识点
- 每个文件标注 `// 【知识点】对应第X章`
- 附带功能与知识点对照表
- 附带扩展练习方向（3-5 个）

### 4.5 P4：专题补充（可选，视模块而定）

- 新版本特性速查手册
- 旧版本兼容内容（如 Vue 2 / TS < 4.0）
- 迁移指南
- 与竞品/替代方案的对比表

## 五、Phase 4：提交推送规范

### 5.1 Commit Message 格式

```
<type>(<scope>): <subject>

<body>
```

**type 取值**：
- `docs`: 新增文档（首次生成）
- `feat`: 新增功能/章节
- `refactor`: 重构/优化现有内容
- `chore`: 格式调整/目录变更

**scope 取值**：模块名缩写（ts/vue/js/react/html/node...）

**示例**：
```
docs(react): 添加 React 基础知识指南与面试题

- React/Basics/React-Basics-Guide.md: 15章完整指南
- React/Interviews/React-Interview-Questions.md: 48道分级面试题
```

优化提交示例：
```
feat(react): React 文档全面优化（5项改进）

P0: 5道核心原理题增加手写实现代码
P1: 关键概念添加流程图（5组）
P2: 16道核心题增加追问链
P3: 新增实战案例：组件库封装
P4: 新增 React 19 新特性速查
```

### 5.2 推送前检查清单

- [ ] 两份文档都已生成并放入正确目录
- [ ] .gitkeep 占位文件已删除（如有内容放入）
- [ ] Git 用户信息正确（`git config user.name/email`）
- [ ] Commit message 符合格式规范
- [ ] 仓库 README 如有模块进度表则同步更新

## 七、快速启动指令

当需要为新模块生成文档时，只需说：

> **「针对 {模块名}，整理出一套基础知识和面试题」**

AI 将自动按本 SOP 执行全部 4 个 Phase。

如果需要跳过优化阶段直接生成初稿，可以说：

> **「针对 {模块名}，先生成初稿，后续再优化」**

如果已有文档需要优化，可以说：

> **「评审并优化 {模块} 的文档」**
